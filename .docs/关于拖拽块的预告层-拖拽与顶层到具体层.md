你说得非常对，**这些确实是硬骨头**。即便底层引擎已经打通，要在 UI 交互层实现完美的“丝滑感”，依然面临三个极其棘手的挑战：

### 硬骨头 1：数据的“细胞级”拆分 (Tab Extraction)
目前的引擎是“整块搬运”（搬运整个 Shell）。
*   **难点**：当你拖拽一个 Tab 标签而不是整个标题栏时，你需要从一个 `TagLeafData` 的 `data` 数组中**抠出一个对象**，然后瞬间为它**克隆一个全新的壳 (Shell)**。
*   **逻辑挑战**：
    *   如果这个标签组本来有 3 个 Tab，抠走 1 个，剩下 2 个还得留在原处。
    *   如果这个标签组只有 1 个 Tab，抠走它后，原来的整个 `Shell` 必须消失。
    *   **这要求引擎具备“原地裂变”的能力**，这比单纯的移动要复杂一个量级。

### 硬骨头 2：几何碰撞与“缝隙检测” (Hit Testing)
这是可视化布局最难的部分。
*   **难点**：当你拖着那个“悬浮窗”在屏幕上晃悠时，系统怎么知道你现在悬停在“侧边栏”和“编辑区”的**那条缝**上？
*   **逻辑挑战**：
    *   你需要建立一套 **“几何映射表”**。在拖拽开始时，扫描所有 `Canvas` 和 `Shell` 的 `getBoundingClientRect()`。
    *   当鼠标移动时，不停地计算：坐标 $(x, y)$ 落在哪个矩形区域？是矩形的左边 20%（代表要在左侧插入）还是中间（代表要合并标签页）？
    *   **性能瓶颈**：这种高频计算不能引起 Vue 的重新渲染，否则拖拽会卡顿。

### 硬骨头 3：状态的“无缝接力” (State Handover)
*   **难点**：你按下的 DOM 元素在 `main-layer`，但执行 `MOVE_NODE` 后，这个 DOM 被销毁了，新 DOM 在 `top-layer` 生成。
*   **逻辑挑战**：
    *   浏览器的原生原生 `mousemove` 事件监听器可能会因为绑定对象的消失而中断。
    *   你必须实现一个 **“全局拖拽单例”**，确保鼠标按下的那一刻，逻辑控制权就交给了 `window`，而不是具体的某个 DOM 元素，这样才能实现从网格中“丝滑剥离”。

关于预告层

关于**预告层（Drop Indicator / Ghost Overlay）**，这是提升布局系统“高级感”和“易用性”的关键。它本质上是一个**独立的、只读的视觉反馈层**，用来告诉用户：“如果你现在松手，卡片就会落在这个深蓝色的区域”。

按照你的架构，预告层不应该修改任何数据，它只是在 `LayoutWorkspace` 中新增的一个**动态视觉指示器**。

以下是实现预告层的深度设计方案：

### 1. 预告层的本质：一个“几何探测器”

预告层不需要复杂的树结构，它只需要一个全局状态来控制显示。

```typescript
// src/renderer/src/features/layout/stores/dragState.ts
export const dropPreview = reactive({
  visible: false,
  x: 0,
  y: 0,
  width: 0,
  height: 0,
  // 记录预判的着陆信息，供 MouseUp 时使用
  targetContainerId: '',
  targetIndex: 0,
  dropType: 'center' as 'left' | 'right' | 'top' | 'bottom' | 'center' | 'none'
})
```

---

### 2. 核心逻辑：五分位探测法（5-Zone Detection）

这是 IDE 布局引擎的经典算法。当你拖拽卡片进入一个 `TagFolder`（容器）的范围时，我们将这个矩形区域划分为 5 个感应区：

*   **中心 (40%)**：合并标签页（进入现有的 Tab 组）。
*   **四周 (各 20%)**：拆分屏幕（在左/右/上/下创建一个新的网格位）。

**探测伪代码：**
```typescript
function calculateDropZone(mouseX, mouseY, rect) {
  const { left, top, width, height } = rect;
  const px = (mouseX - left) / width; // 鼠标在矩形内的横向百分比
  const py = (mouseY - top) / height; // 纵向百分比

  if (px < 0.2) return 'left';
  if (px > 0.8) return 'right';
  if (py < 0.2) return 'top';
  if (py > 0.8) return 'bottom';
  return 'center';
}
```

---

### 3. 组件实现：`DropIndicator.vue`

这个组件平时是隐藏的，只有当 `dropPreview.visible` 为真时，才会在 `LayoutWorkspace` 的最顶层渲染出来。

```vue
<!-- src/renderer/src/features/layout/views/DropIndicator.vue -->
<template>
  <div 
    v-if="dropPreview.visible"
    class="drop-indicator"
    :style="{
      left: dropPreview.x + 'px',
      top: dropPreview.y + 'px',
      width: dropPreview.width + 'px',
      height: dropPreview.height + 'px'
    }"
  >
    <div class="drop-indicator-inner"></div>
  </div>
</template>

<style scoped>
.drop-indicator {
  position: absolute;
  z-index: 9999;
  pointer-events: none; /* 绝对不能阻挡鼠标事件 */
  transition: all 0.15s ease-out; /* 让蓝色方块滑动的丝滑感 */
}
.drop-indicator-inner {
  width: 100%;
  height: 100%;
  background: rgba(0, 120, 212, 0.3); /* 经典的淡蓝色 */
  border: 2px solid #0078d4;
  box-sizing: border-box;
}
</style>
```

### 4. 骨头在哪？（难点分析）

预告层的**难点不在于显示，而在于如何高效地“感知”下方的容器。**

1.  **DOM 注册机制**：
    由于你的布局是递归嵌套的，预告层无法直接通过数据知道每个 `div` 在屏幕上的物理位置。你需要让每个 `TagFolder` 在 `onMounted` 时，将其 `id` 和 `DOM 引用` 注册到一个全局的 `Map` 中。

2.  **高频碰撞计算**：
    在 `mousemove` 期间，你需要遍历这个 `Map`，寻找鼠标当前所在的容器。
    *   *挑战*：如果有容器重叠（比如自由层覆盖在网格层上），你必须优先匹配 Z-index 最高的那个容器。

3.  **坐标转换**：
    如果用户拖拽的是一个 Tab，预告层不仅要显示“拆分”的方块，还要能预判插入后的 `ratio`。例如，如果感应到 `left` 区，预览方块应该只占目标容器左侧的 50%。

它应该是 `LayoutWorkspace.vue` 里的最后一个元素：

```vue
<template>
  <div class="stack-wrapper">

    <!-- 永远在最顶层，作为视觉引导 -->
    <DropIndicator />
  </div>
</template>
```

### 总结预案：

预告层是你那套底层引擎的 **“翻译官”**。
*   它观察用户的鼠标动作。
*   它把模糊的坐标 $(x, y)$ 翻译成具体的 `MOVE_NODE` 参数（`targetContainerId`, `index`, `ratio`）。
*   当用户松手时，它把这些精确的参数喂给 `dispatch`。

**这确实是硬骨头，因为它要求你从“纯数据思维”切换到“几何碰撞思维”。** 但只要这一层做好了，你的软件就会从“能用”变成“好用”。

预告层分为两个部分，一部分是缝隙，另一个部分是块。缝隙就是指两个容器之间的缝，它通常是指同一个容器组合中的缝隙。而块的意思是，如果你试图把一个块直接拖到另一个块的上面，那你没拖到缝上面，我就不知道你要拖到哪个位置。这时候我会根据你拖到这个块的上下左右来决定我要放在哪个位置。比如，如果它是一个横向的容器，那我拖到左边，左半部分那就往左放，右半部分那就放在这个容器的右，往右放。下半部分就看这，如果它放在下半部分，那我要判断这玩意是不是竖向容器。如果不是一个竖向容器，那我没办法放在下面，那就直接把它切成左右两半，你放下面也没有任何效果。除了左右以外，还有中间部分。如果你放在中间部分，不左不右，最中间的位置。那我就会认为你试图把这个标签插入到这个标签组中，因为一个标签组中有多个标签，那你是插到里面，不作为横向或竖向的容器，只是作为标签组中的一个标签。

你的这个想法非常深刻，你准确地描述了**现代桌面级布局系统（如 VS Code, IntelliJ 或 Chrome）在处理 Docking 时的核心交互逻辑**。

你把预告层拆分为“缝隙”和“块”两个部分，本质上是在处理两种不同的逻辑意图：**“排序（Reordering）”**与**“分割/合并（Splitting/Merging）”**。

这根骨头的硬度在于，它要求你的 `Coordinator` 和 `Engine` 具备**“动态重组树结构”**的能力。

---

### 1. “缝隙”逻辑：同级排序 (Insertion)
当鼠标移动到两个卡片之间的狭长地带（Gap）时，预告层显示为一条**蓝色的粗线**。
*   **物理意义**：用户想在现有的 `Canvas` 数组中，在 `Index A` 和 `Index B` 之间插入一个新成员。
*   **操作**：调用 `MOVE_NODE`，传入确定的 `targetContainerId`（当前的 Canvas ID）和计算出的 `index`。

---

### 2. “块”逻辑：空间分割与堆叠 (Split & Stack)
这是最复杂的。当你把卡片拖到一个现有的“块（Shell）”上面时，根据落点，会有三种完全不同的结果：

#### A. 中间部分：堆叠 (Stacking)
*   **显示**：整个块中心区域覆盖一层半透明蓝光。
*   **物理意义**：**“合并标签页”**。用户不想改变排版，只想把这个 Tab 塞进这个标签组。
*   **操作**：这不属于 `MOVE_NODE` 移动壳，而是 `MOVE_TAB` 移动原子数据。

#### B. 同向边缘：插入 (Appending)
*   **场景**：父容器是**横向**的，你拖到了当前块的**最左边**或**最右边**。
*   **物理意义**：用户想在当前块的旁边增加一个新的块。
*   **操作**：直接在父 `Canvas` 的数组里，在当前块的前面或后面 `splice` 进去。

#### C. 正交边缘：劈裂 (Splitting) —— **这是难点中的难点**
*   **场景**：父容器是**横向**的，但你拖到了当前块的**下半部分**。
*   **物理意义**：用户想把这个位置**垂直一分为二**。
*   **操作**：
    1.  你不能直接往父 `Canvas` 里插，因为父 Canvas 是横向的，插进去只会导致水平并排。
    2.  **树结构重组**：你需要创建一个新的、**纵向**的 `Canvas`。
    3.  把“原本在那里的块”和“新拖进来的块”一起塞进这个新 `Canvas`。
    4.  最后把这个新 `Canvas` 替换掉原本那个块在父级数组中的位置。

---

### 3. 你提到的“没效果”情况 (Direction Constraint)
你提到：*“如果它不是一个竖向容器，那我没办法放在下面……放下面也没有任何效果。”*

其实在高级布局中，这不叫“没效果”，而是一个**交互抉择**：
*   **方案一（如你所说，保守型）**：禁用不符合当前方向的落点，强制用户必须先找到“缝隙”。
*   **方案二（VS Code 方案，激进型）**：**自动变向**。只要用户拖到了下边，系统就认为用户想把这里变成垂直排版，从而自动生成一个新的子 Canvas 来承载。

---

### 4. 架构设计的落地建议

为了支持你说的“块”与“缝隙”预告，你的 `DropIndicator.vue` 需要具备两套皮肤：

1.  **皮肤 A (Line)**：用于“缝隙”，`width: 4px`，`height: 100%`。
2.  **皮肤 B (Rect)**：用于“块”，跟随鼠标落点动态改变 `left/right/top/bottom/center` 的宽度。

**实现预判的逻辑链路：**
1.  **探测器**：在拖拽时，寻找鼠标下方的 `DOM` 元素，通过 `getAttribute('id')` 拿到容器 ID。
2.  **逻辑判断**：
    *   如果是 `Canvas` 边缘 $\to$ 设置预告为 **“线”**。
    *   如果是 `Shell` 内部 $\to$ 计算五分位 $\to$ 设置预告为 **“方块”**。
3.  **结果输出**：
    *   预告层不仅显示颜色，还会实时计算出 `Target ID`、`Action Type`（是 Move 还是 Split）以及 `Index`。

### 总结
这套逻辑做下来，你的布局系统将具备**“自适应重组”**的能力。**“缝隙”是操作数组，而“块”是操作树的分叉。**

这确实是硬骨头，尤其是**正交劈裂（Split）**时的树结构变换。但请放心，你现在的 `insertNodeIntoTree` 和 `walkTree` 已经提供了足够的灵活性来支持这种变换。

# 布局系统预告层（Drop Indicator）交互设计规范

## 1. 核心定位
预告层是位于所有布局层之上的**虚拟视觉反馈层**。它的唯一职责是：在用户拖拽卡片（Shell）过程中，实时捕捉鼠标坐标与下方容器（Canvas/Shell）的几何关系，并将即将发生的“数据变更结果”以视觉化的方式提前呈现给用户。

---

## 2. 预告的两大形态
根据鼠标落点的不同，预告层呈现为两种截然不同的形态：**“缝隙线”** 与 **“占位块”**。

### 2.1 缝隙预告（The Gap Line）
*   **触发场景**：当鼠标移动到两个相邻卡片（Shell）之间的边缘区域，或容器（Canvas）的起始/末尾边缘时。
*   **视觉表现**：呈现为一根具有厚度的**高亮蓝色长线**。
*   **操作意图**：**线性重排（Reordering）**。
*   **用户心理**：用户希望将卡片“插入”到两个既有卡片之间，而不改变现有的排版方向和结构。

### 2.2 块级预告（The Block Region）
*   **触发场景**：当鼠标直接悬停在某个具体的卡片（Shell）上方时。
*   **视觉表现**：呈现为一个**半透明的蓝色矩形色块**。
*   **操作意图**：**空间分割（Splitting）** 或 **内容合并（Merging）**。
*   **用户心理**：用户希望对当前这个“块”所占据的空间进行重新分配，或者将内容合并进去。

---

## 3. 块级预告的五分位感应逻辑
当预告层识别到鼠标处于一个“块（Shell）”上方时，会将该矩形区域划分为五个感应区。落点不同，预告块的大小和位置会动态变化：

### ① 中心区域（Center Zone）
*   **感应位置**：卡片的中心矩形范围内。
*   **预告表现**：蓝色色块覆盖整个目标卡片区域。
*   **业务目标**：**标签合并**。表示松开鼠标后，当前拖拽的标签页将作为“新成员”加入到目标标签组中。

### ② 同向边缘（Left / Right for Horizontal）
*   **感应位置**：卡片的左边缘或右边缘。
*   **预告表现**：蓝色色块仅占据目标卡片的左侧或右侧（例如 50% 宽度）。
*   **业务目标**：**同级水平拆分**。在当前水平序列中，于目标卡片的侧面挤入一个新的坑位。

### ③ 正交边缘（Top / Bottom for Vertical）
*   **感应位置**：卡片的上边缘或下边缘。
*   **预告表现**：蓝色色块仅占据目标卡片的上半部或下半部。
*   **业务目标**：**垂直拆分/变向插入**。
*   **方向约束逻辑**（关键细节）：
    *   **匹配情况**：如果该容器属于一个“纵向排布”的父容器，预告块正常显示，表示在这里增加一个上下层级。
    *   **冲突情况**：如果当前区域属于一个“横向排布”的容器且没有嵌套纵向容器。
        *   *用户抉择*：若用户强制拖到下半部分，系统需判断是否允许在此处临时生成一个“纵向嵌套容器”。
        *   *预设规则*：若不满足方向兼容性且系统未开启自动变向，则预告层**不显示**或**无效果**，提示用户此落点无效。

---

## 4. 交互语义总结

| 用户动作（落点） | 预告形态 | 系统理解的意图 | 最终结果 |
| :--- | :--- | :--- | :--- |
| 两个卡片的夹缝 | **线 (Line)** | “我想插在他们中间” | 在数组指定 Index 插入新 Shell |
| 卡片的正中心 | **全覆盖块 (Full Rect)** | “我想把这两个组并在一起” | 提取 Tab 数据合并至目标 Leaf |
| 卡片的左右边缘 | **侧边半块 (Side Rect)** | “我想在旁边开个分屏” | 在水平 Canvas 中插入新 Shell |
| 卡片的上下边缘 | **上下半块 (Half Rect)** | “我想在上下开个分屏” | 插入新 Shell (受限于容器方向) |

---

## 5. 设计目标
*   **确定性**：用户在松手前，必须通过预告层 100% 确定卡片将要去往的位置。
*   **零侵入**：预告层只是一层“幻影”，在计算碰撞时，它不能对底层的业务逻辑产生任何副作用。
*   **实时性**：随着鼠标的毫秒级移动，预告层必须流畅地在“线”和“块”之间切换，不能有视觉抖动。

# 布局系统预告层（Drop Indicator）交互设计规范

## 1. 核心定位
预告层是位于所有布局层之上的**虚拟视觉反馈层**。它的唯一职责是：在用户拖拽卡片（Shell）过程中，实时捕捉鼠标坐标与下方容器（Canvas/Shell）的几何关系，并将即将发生的“数据变更结果”以视觉化的方式提前呈现给用户。

---

## 2. 预告的两大形态
根据鼠标落点的不同，预告层呈现为两种截然不同的形态：**“缝隙线”** 与 **“占位块”**。

### 2.1 缝隙预告（The Gap Line）
*   **触发场景**：当鼠标移动到两个相邻卡片（Shell）之间的边缘区域，或容器（Canvas）的起始/末尾边缘时。
*   **视觉表现**：呈现为一根具有厚度的**高亮蓝色长线**。
*   **操作意图**：**线性重排（Reordering）**。
*   **用户心理**：用户希望将卡片“插入”到两个既有卡片之间，而不改变现有的排版方向和结构。

### 2.2 块级预告（The Block Region）
*   **触发场景**：当鼠标直接悬停在某个具体的卡片（Shell）上方时。
*   **视觉表现**：呈现为一个**半透明的蓝色矩形色块**。
*   **操作意图**：**空间分割（Splitting）** 或 **内容合并（Merging）**。
*   **用户心理**：用户希望对当前这个“块”所占据的空间进行重新分配，或者将内容合并进去。

---

## 3. 块级预告的五分位感应逻辑
当预告层识别到鼠标处于一个“块（Shell）”上方时，会将该矩形区域划分为五个感应区。落点不同，预告块的大小和位置会动态变化：

### ① 中心区域（Center Zone）
*   **感应位置**：卡片的中心矩形范围内。
*   **预告表现**：蓝色色块覆盖整个目标卡片区域。
*   **业务目标**：**标签合并**。表示松开鼠标后，当前拖拽的标签页将作为“新成员”加入到目标标签组中。

### ② 同向边缘（Left / Right for Horizontal）
*   **感应位置**：卡片的左边缘或右边缘。
*   **预告表现**：蓝色色块仅占据目标卡片的左侧或右侧（例如 50% 宽度）。
*   **业务目标**：**同级水平拆分**。在当前水平序列中，于目标卡片的侧面挤入一个新的坑位。

### ③ 正交边缘（Top / Bottom for Vertical）
*   **感应位置**：卡片的上边缘或下边缘。
*   **预告表现**：蓝色色块仅占据目标卡片的上半部或下半部。
*   **业务目标**：**垂直拆分/变向插入**。
*   **方向约束逻辑**（关键细节）：
    *   **匹配情况**：如果该容器属于一个“纵向排布”的父容器，预告块正常显示，表示在这里增加一个上下层级。
    *   **冲突情况**：如果当前区域属于一个“横向排布”的容器且没有嵌套纵向容器。
        *   *用户抉择*：若用户强制拖到下半部分，系统需判断是否允许在此处临时生成一个“纵向嵌套容器”。
        *   *预设规则*：若不满足方向兼容性且系统未开启自动变向，则预告层**不显示**或**无效果**，提示用户此落点无效。

---

## 4. 交互语义总结

| 用户动作（落点） | 预告形态 | 系统理解的意图 | 最终结果 |
| :--- | :--- | :--- | :--- |
| 两个卡片的夹缝 | **线 (Line)** | “我想插在他们中间” | 在数组指定 Index 插入新 Shell |
| 卡片的正中心 | **全覆盖块 (Full Rect)** | “我想把这两个组并在一起” | 提取 Tab 数据合并至目标 Leaf |
| 卡片的左右边缘 | **侧边半块 (Side Rect)** | “我想在旁边开个分屏” | 在水平 Canvas 中插入新 Shell |
| 卡片的上下边缘 | **上下半块 (Half Rect)** | “我想在上下开个分屏” | 插入新 Shell (受限于容器方向) |

---

## 5. 设计目标
*   **确定性**：用户在松手前，必须通过预告层 100% 确定卡片将要去往的位置。
*   **零侵入**：预告层只是一层“幻影”，在计算碰撞时，它不能对底层的业务逻辑产生任何副作用。
*   **实时性**：随着鼠标的毫秒级移动，预告层必须流畅地在“线”和“块”之间切换，不能有视觉抖动。

## 6. 预告层与插入逻辑的统一性思考

### 核心洞察

在思考预告层的实现时，我发现了一个重要的架构问题：**预告层不应该只是一个视觉反馈组件，它实际上应该承担"插入决策"的责任**。

这个洞察来自于对两个操作的观察：
1. **预告操作**：用户拖拽时，系统需要决定"在哪条缝隙处显示线条"或"在哪个块区域显示色块"
2. **插入操作**：用户松手时，系统需要决定"将对象插入到哪个位置"

这两个操作本质上是在做同样的事情：**找到一个位置，然后在那个位置插入对象**。

### 为什么需要统一

如果我们让预告层和最终插入使用两套不同的算法，就会出现严重的问题：

1. **不一致性风险**：预告层显示的位置和实际插入的位置可能不一致
   - 预告层算法可能认为应该在索引2的位置插入
   - 实际插入算法可能计算出应该在索引3的位置插入
   - 用户看到的是一条线，但松手后对象却落在了另一个地方

2. **维护成本**：两套算法意味着两套逻辑需要同时维护
   - 当布局逻辑变化时，需要同步修改两处代码
   - 容易出现一处修改了但另一处忘记更新的情况

3. **理解困难**：代码中存在两个相似的算法会让人困惑
   - 开发者需要理解为什么有两套算法
   - 新加入的团队成员可能会被不同的实现搞混

### 统一后的架构

**预告层的职责扩展**：
- 传统职责：根据鼠标位置计算并显示预览（线条或色块）
- 新增职责：在显示预览的同时，记录"如果现在松手，应该插入到哪个位置"

**插入操作的简化**：
- 当用户松手时，不需要重新计算插入位置
- 直接读取预告层已经计算好的位置信息
- 执行插入操作即可

### 具体实现思路

在 DropIndicator.vue 中，当计算出预览位置时，同时记录插入决策：

```typescript
// 在 DropIndicator.vue 中
const dropDecision = computed(() => {
  if (!dropPreview.visible) return null
  
  // 这里的算法同时服务于两个目的：
  // 1. 决定预览线/块显示在哪里
  // 2. 决定如果松手，应该插入到哪里
  
  return {
    targetContainerId: '...',  // 目标容器ID
    targetIndex: 2,            // 插入索引
    dropType: 'gap'            // 插入类型（缝隙或块）
  }
})
```

在拖拽结束（mouseup）时，直接使用这个决策：

```typescript
// 在 layoutCoordinator.ts 中
function handleDrop() {
  const decision = dropDecision.value
  if (decision) {
    // 直接使用预告层已经计算好的决策
    executeInsertion(decision)
  }
}
```

### 关键优势

1. **所见即所得**：用户看到的预览位置就是最终的插入位置，不会有任何偏差
2. **单一真理来源**：只有一个算法负责位置计算，避免了不一致
3. **代码简洁**：减少了重复逻辑，降低了维护成本
4. **易于调试**：如果预览位置不对，修复预览算法就会自动修复插入问题

### 可能的挑战

1. **状态管理**：需要确保 dropDecision 在合适的时机更新，并且在松手时能够正确读取
2. **性能考虑**：需要保证计算不会因为额外的状态存储而变慢
3. **边界情况**：需要处理鼠标快速移动、快速松手等边缘情况

### 总结

预告层不应该只是一个"视觉花瓶"，它应该是整个拖拽插入系统的"大脑"。它负责理解用户的意图（通过鼠标位置），计算出最佳插入位置，同时告诉用户"我会把它插在这里"（视觉预览）和告诉系统"真的要把它插在这里"（插入决策）。这样设计既保证了用户体验的一致性，又简化了系统架构。

## 7. 补充思考：实施细节和边界情况

### 7.1 时机和状态的可靠性

**"最后预览状态"的捕获机制**
- 需要特别关注"最后预览状态"的捕获时机
- 如果用户快速移动鼠标后立即松手，必须确保执行的是"最后一次mousemove计算出的位置"，而不是"较早计算出的位置"
- 这可能需要在松手时做一个"快照"操作，将当前的dropDecision保存下来，避免异步更新导致的偏差

**建议的实现方式**：
```typescript
// 在 mouseup 时立即创建快照
function handleMouseUp() {
  const snapshot = { ...dropDecision.value }  // 立即创建快照
  if (snapshot) {
    executeInsertion(snapshot)  // 使用快照执行插入
  }
}
```

### 7.2 边界情况的处理策略

**鼠标移出有效区域的情况**
- 当鼠标移出有效区域时，预览消失，此时如果松手应该如何处理？
  - 选项A：取消拖拽，不执行插入（更安全，但可能误操作）
  - 选项B：使用"最后一次有效位置"（更符合用户预期，但需要处理有效性判断）
- 建议采用选项B，因为用户拖拽后移出区域再松手，通常还是期望插入到最后一个有效位置

**dropDecision计算失败的情况**
- 如果dropDecision计算失败或为null，系统应该如何应对？
  - 提供明确的用户反馈（比如显示"无效位置"的提示）
  - 或者取消拖拽操作，让用户重新尝试

### 7.3 性能优化的考虑

**高频计算的优化策略**
- 既然计算位置是高频操作（每次mousemove都会触发），需要考虑性能优化
- 可以使用防抖或节流技术来限制计算频率
- 或者只在位置变化超过一定阈值时才重新计算，避免微小移动导致的无效计算

**建议的实现方式**：
```typescript
// 使用节流限制计算频率
const throttledCalculateGap = useThrottleFn(calculateGapPosition, 16)  // 16ms ≈ 60fps

function handleMouseMove(e: MouseEvent) {
  throttledCalculateGap(e.clientX, e.clientY)
}
```

### 7.4 可测试性优势

**这种统一架构的一个好处是更容易进行单元测试**
- 可以直接测试"给定鼠标位置，计算出什么插入决策"
- 不需要依赖复杂的拖拽交互就能验证算法的正确性
- 测试用例示例：
  ```typescript
  describe('dropDecision calculation', () => {
    it('should calculate correct gap position between two containers', () => {
      // 模拟鼠标位置和容器布局
      const result = calculateDropDecision(mouseX, mouseY, containers)
      expect(result.targetIndex).toBe(2)
      expect(result.dropType).toBe('gap')
    })
  })
  ```

### 7.5 向后兼容性考虑

**如果将来需要支持不同的插入算法**
- 比如支持"智能插入"（AI推荐的最佳位置）和"精确插入"（用户指定的精确位置）
- 可以通过策略模式来实现，让dropDecision根据不同的策略返回不同的结果
- 这样可以在不改变整体架构的情况下扩展功能

**建议的实现方式**：
```typescript
type InsertStrategy = 'precise' | 'smart' | 'none'

const dropDecision = computed(() => {
  if (!dropPreview.visible) return null
  
  switch (insertStrategy.value) {
    case 'precise':
      return calculatePrecisePosition(mousePos, containers)
    case 'smart':
      return calculateSmartPosition(mousePos, containers)
    default:
      return null
  }
})
```

### 7.6 用户体验的细节优化

**过渡动画的考虑**
- 当预览位置快速切换时，是否需要添加过渡动画？
- 线条和色块的切换是否应该有平滑的过渡效果？
- 这些细节可以提升用户体验，但需要权衡性能开销

**视觉反馈的增强**
- 除了显示线条/色块，是否还需要其他视觉提示？
  - 比如：当检测到有效插入位置时，目标容器的边框高亮
  - 或者：显示一个小的工具提示，告诉用户"松手后将插入到这个位置"
- 这些增强可以帮助用户更好地理解系统当前的状态
