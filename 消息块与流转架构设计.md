# 消息块架构与聊天终端定位设计文档

**文档版本**：v1.0
**状态**：架构敲定
**核心议题**：明确“聊天面板”在多智能体工作流系统中的终极定位，以及规范前后端消息块（Message Block）的统一数据结构与流转机制。

---

## 一、 核心哲学：聊天页面的“终极终端”定位

在设计前后端的数据交互契约之前，必须首先解答一个根本性的产品架构问题：**在包含节点图、智能体编排、编辑器等复杂元素的 IDE 环境中，聊天页面的作用到底是什么？它是唯一的交互入口吗？**

### 1. 原始的疑惑与推演
起初的设想中，如果系统拥有强大的节点工作流（Workflow），用户似乎应该在多个节点中并行输入指令（例如：给 Agent A 输入一段话，同时给 Agent B 输入另一段话，然后让它们协同工作）。如果真是这样，单一的聊天页面就无法作为唯一的输入口。

### 2. 最终的觉悟：工作流是“大脑”，聊天框是“显示器”
经过深思熟虑，我们确立了系统的核心运行逻辑：
* **工作流面板（Node Graph）的本质是“后台逻辑与大脑结构”**：工作流页面仅仅是用来**编写和定义** AI 是如何思考的、调用的流程是什么、会触发哪些子 Agent。
* **运行时的不可变性**：内部的中间流程、中间提示词都在工作流页面里编辑，但它们是静态的定义。**一旦工作流开始运行，用户在流程图里的中途修改不会对当前运行产生任何影响**，修改只能在下一次运行中生效。这才是符合计算机科学常理的设计。
* **聊天页面的终极定位**：基于上述逻辑，无论背后的工作流有多复杂、并行了多少个 Agent，它们最终的输出流、思考过程、工具调用状态，都必须汇聚到一个地方展示。因此，**聊天页面（Chat Panel）绝对不是一个简单的对话框，它是整个 AI 操作系统的“终极显示终端（Terminal/Console）”**。

## 二、 前端架构：超大联合类型与 Block 渲染机制

既然确立了聊天页面是“终极显示器”，那么前端页面的实现逻辑就变得极其清晰：

### 1. 视图层的本质：由上至下的 Block 列表
前端的聊天页面非常好做，它本质上就是一个单纯的列表容器。无论后端的工作流经历了怎样复杂的推演，反映在页面上就是**一个接一个不断生成的 Block（消息块）**，从上往下持续渲染。

### 2. 动态组件映射（1 Block = 1 Vue Component）
前端不需要写复杂的条件判断，而是采用高度解耦的组件化策略：
* 用 Vue 根据数据结构动态生成组件。
* 每一个数据块（Block）严格对应一个特定的 Vue 组件（例如 `CodeBlock.vue`、`ThinkingBlock.vue`、`ToolCallBlock.vue`）。
* 前端只需要负责把这些组件“组装”在列表里即可。

### 3. 数据结构：TypeScript 超大联合类型 (Discriminated Union)
为了支撑这种灵活的显示，我们需要定义一个能够容纳所有可能情况的数据结构。
它本质上是一个**数组**，数组内包含一个**超大的联合类型（Message Union Type）**。

```typescript
// 理论上的数据结构演示
type BlockType = 'text' | 'code' | 'thinking' | 'tool_call' | 'agent_handoff' | 'error';

// 基础类型
interface BaseBlock {
  id: string;
  type: BlockType;
}

// 具体的消息块类型（每一个类型内部的数据结构都不一样）
interface TextBlock extends BaseBlock { type: 'text'; content: string; }
interface CodeBlock extends BaseBlock { type: 'code'; code: string; language: string; }
interface ThinkingBlock extends BaseBlock { type: 'thinking'; content: string; isFinished: boolean; }
interface ToolCallBlock extends BaseBlock { type: 'tool_call'; toolName: string; args: any; status: string; }

// 终极联合类型：消息数组里装的就是这个类型
export type UnifiedMessageBlock = TextBlock | CodeBlock | ThinkingBlock | ToolCallBlock;
```
**优势**：通过这种设计，未来无论我们增加了什么新奇的 Agent 能力，只需要在这个联合类型里扩充一个新的类型，并在前端加一个对应的 Vue 组件，就可以自由地在页面上展示，核心列表逻辑一行都不用改。

## 三、 后端架构：防腐层与内部真理模型

前端的联合类型是为了“方便显示”，但这个数据必须由后端提供。这就引出了最核心的前后端契约问题：**后端到底传什么给前端？后端又是如何处理这些消息的？**

后端不仅要和前端交互，还要和各种不同厂商的大语言模型（LLM API）交互。不同厂商（OpenAI, Anthropic, DeepSeek）的返回格式、通用标准各不相同。为了系统的稳健，我们必须在后端建立**严格的分层与兼容转换机制**。

### 1. 第一层：外部 API 实体层 (External API Layer)
这一层专门用于接收各大 LLM 厂商的原始消息，定义其特有的数据结构。
由于每个 API 不一样，系统需要一个基础的大类（API LLM 基类），然后细分判断具体是哪一种厂商的架构（OpenAI 格式、特定厂商扩展格式等）。**这一层的数据是“脏”的、不可控的。**

### 2. 第二层：兼容转译层 (Adapter / Anti-Corruption Layer)
这是后端最关键的“防腐层”。
无论第一层收到的是什么奇形怪状的数据结构，只要确认它是从 API 来的，**兼容层就会将其强制转译（翻译）成我们系统内定的、统一的内部数据结构。**

### 3. 第三层：内部核心业务层 (Internal Domain Layer)
这是我们自己的“内部层”。**系统的核心代码逻辑（也就是业务逻辑）绝对不关心外部大模型 API 的原始数据结构是什么样子的。**
核心业务层只认识经过转译后的“内部数据结构”。所有的内部计算，包括：
* 注入系统提示词
* 调度不同的 Agent
* 解析并执行工具调用（Tool Calling）
* 工作流的节点数据流转

全部基于这套内部类型进行操作。

### 4. 第四层：前端契约层 (Frontend Contract)
当内部层完成了所有的业务计算（或者在流式计算的中途），它会将内部的数据结构序列化，作为最终的“消息块（Block）”传递给前端。前端的 TypeScript 联合类型，完美对应了后端的这套内部数据结构。

## 四、 总结：数据的完整生命周期

梳理整个系统的消息处理闭环，其链路如下：

1. **[LLM 端]** 大模型吐出带有厂商特定格式的 JSON 流。
2. **[后端 - API层]** 接收到特定的原始数据结构。
3. **[后端 - 兼容层]** 剥离厂商特性，将其转译为系统通用的 `InternalBlock`（如：判定这是一个工具调用请求）。
4. **[后端 - 业务层]** 使用 `InternalBlock` 执行本地计算机的实际操作（如读取文件、调用子 Agent），并产生新的状态（如：工具执行中、思考中）。
5. **[网络传输]** 后端将标准的 `MessageBlock` 数据流（包含具体的 `type` 和 `data`）推送到前端。
6. **[前端 - 逻辑层]** 聊天页面（终极终端）接收到联合类型的数组。
7. **[前端 - 视图层]** Vue 根据 `type` 字段，动态渲染出一个个独立的 Block UI 组件，最终在用户眼前呈现出一个结构清晰、内容丰富的 AI 运行全纪录。
