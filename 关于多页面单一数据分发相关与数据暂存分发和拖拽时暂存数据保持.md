一、 Photoshop 是怎么存数据的？
你说得对，新建图层没保存前，数据就在内存里。但它的实现方式比“存一个变量”要复杂且严谨得多。

内存模型：
Photoshop 本质上是一个 C++ 程序。当你新建一个图层，它会在堆内存中开辟一块空间，存储这个图层的所有属性（宽、高、像素数据Buffer、混合模式、透明度等）。
它不是存一个简单的变量，而是维护了一个文档对象模型。这就好比一颗大树，根节点是“文档”，树枝是“图层组”，叶子是“图层”。
像素数据通常以压缩格式或原始缓冲区的形式存在内存中。
软持久化：
Photoshop 确实做了“软持久化”，但这主要是为了崩溃恢复和历史记录。
历史记录： 每一步操作都会把当前状态（或差异）序列化存起来。这不是存到硬盘文件，而是存到内存或暂存盘，为了让你能 Ctrl+Z。
自动保存： 后台定时把内存数据序列化写到磁盘临时文件。
结论： 在用户点击“保存”之前，数据的主阵地永远是内存。它不需要像Web开发那样一变动就往数据库存，那样性能会爆炸。

二、 你的 Vue 困境：为什么你会觉得“难受”？
你现在的痛点在于：你试图用 Vue 的“组件状态”去承载“业务模型”。

你遇到的死循环逻辑：

我要改页面位置 -> 就要改 Vue Data -> Vue Data 变了页面就重渲染 -> 页面重渲染可能导致组件销毁重建 -> 组件里的临时输入数据丢了。

Photoshop 为什么没这个问题？
因为 Photoshop（包括 Word、Excel 这种重客户端软件）采用的是 MVC（Model-View-Controller） 或 MVVM 架构，且 View 是惰性的。

Model（数据层）： 图层列表、像素数据。它是核心，不管界面怎么变，数据就在那儿。
View（视图层）： 界面只是数据的“投影”。当你拖动图层位置，Photoshop 只是更新了 View 的绘制坐标，或者更新了 Model 里的 order 字段，然后 View 重新读取 Model 画一遍。
Vue 的问题（或者说是误用）：
Vue 太方便了，导致开发者经常把 Data 和 Model 混为一谈。

Vue Data 是为了驱动 UI 渲染存在的。
业务 Model 是为了描述业务逻辑存在的。
如果你把“图层里的输入框内容”直接写在 Vue 组件的 data() 里，当父组件因为位置变动而 key 变了，或者列表重排，Vue 就会销毁这个组件实例，数据自然就丢了。

三、 解决方案：如何像开发 Photoshop 一样开发你的应用？
你需要做架构分层。不论你用 Electron + Vue 还是纯 Web，核心思想都是：数据与视图分离。

1. 建立独立的“数据模型层”
不要让 Vue 管理“核心业务数据”。你需要创建一个单例的类或对象，在内存中长期驻留。

假设你用 TypeScript/JavaScript 开发：


// model/Document.js
// 这是一个纯 JS 对象，完全脱离 Vue 组件生命周期
class Document {
  constructor() {
    // 这里的数据就是你的“图层”数据
    this.layers = []; 
  }

  addLayer(layer) {
    this.layers.push(layer);
    // 通知订阅者（Vue组件）数据变了
    this.emitChange(); 
  }

  updateLayerContent(layerId, newContent) {
    const layer = this.layers.find(l => l.id === layerId);
    if (layer) {
      layer.content = newContent;
      this.emitChange();
    }
  }
}

// 全局单例，只要网页不刷新，它就在内存里
export const currentDocument = new Document();
2. Vue 只作为“视图投影”
你的 Vue 组件里，data 不存业务数据，只存 UI 状态（比如“当前是不是正在拖拽”）。业务数据从 Model 里读取。


// LayerCard.vue

3. 解决“页面刷新”问题（持久化策略）
你担心的“页面不小心刷新了”或者“Vue 热重载了”，这确实是个问题。

对于 Photoshop 级别的复杂度，数据存哪？
实时状态（内存）： 上文提到的 Document 类。这是最快的，操作都在内存。
软持久化：
因为你是 Electron 应用，你有两个极佳的选择：
IndexedDB / LocalStorage： 每次 Model 变动，防抖后存入 IndexedDB。
Electron 主进程文件： 通过 IPC 把数据发给 Node 端，Node 端写到一个临时 JSON 文件里（类似 PS 的暂存盘）。
建议架构：

Vue 页面（渲染进程）： 负责 UI 交互。用户的每一次输入，立刻更新内存中的 Document 对象。
内存对象（渲染进程）： 维护着完整的图层树、数据结构。
自动保存（后台）： 监听 Document 变化，每隔 5 秒或 10 次操作，把 Document 序列化存到 IndexedDB 或 Electron 本地文件。
4. 关于你的“双后端”纠结
你有 C# 后端，又有 Electron Node 后端。存谁？

暂存： 存前端。
理由：用户体验。Photoshop 画一笔就要存后端，那延迟是不可接受的。所有的“图层排布”、“输入框文字”都应优先在内存中维护。
自动保存： 存 Electron Node 后端（本地文件）。
理由：防止浏览器渲染进程崩溃导致数据丢失。Electron 的 Node 端比渲染进程稳定，适合写临时文件。
最终保存： 存 C# 后端（数据库/服务器）。
理由：用户点击“保存”按钮，或者关闭软件时，将内存数据同步到远程服务器。

Vue 的思想是数据驱动页面，但不要把“业务数据”混淆为“组件状态”。
不要让 Vue 组件承担数据管理的责任。 抽离出一个全局的、独立于 Vue 生命周期的 Store（状态管理） 或 Model（模型）。

一、 先解决“内存两份”的焦虑
你说“用户的电脑要存两坨数据”，实际上：

数据量其实没那么大：

二、 再解决“同步灾难”：换个思路，你就豁然开朗
你最怕的是：前端改了，后端没改，两边不一致，还得写代码同步，还要处理失败回滚。

解决之道：把“数据同步”变成“指令分发”。

不要把前端当成“数据源”，把前端当成“遥控器”。

错误的模式（数据同步模式）：
用户输入文字 -> 前端 Data 变了。
发送新 Data 给后端。
后端保存。
灾难点：前端 Data 和后端 Data 不一致怎么办？我要不要等后端返回才渲染？
正确的模式（指令分发模式 —— 类似 Vim / Figma）：
用户输入文字。
前端立刻发出一个指令：Command: { type: 'input', id: 'layer1', value: '你好' }。
关键点：前端发完指令后，不改变自己的状态，或者只做临时的“乐观显示”。
C# 后端 收到指令 -> 更新核心数据 -> 广播一个事件：Event: { type: 'updated', data: {...} }。
前端 监听到这个事件 -> 这才是前端唯一的数据来源。
在这种模式下：

前端依然是无状态的：它不负责“产生数据”，它只负责“展示后端广播回来的数据”。
不需要双向同步：数据流向是单向的。用户操作 -> 指令 -> 后端 -> 广播 -> 前端状态更新 -> 界面更新。
解决同步灾难：前端不需要管有没有同步上，它只需要管：“后端告诉我啥，我就显示啥”。如果网络断了，后端没广播回来，前端界面上这个字就没变（或者显示一个加载失败的图标），这非常符合逻辑。
三、 针对“输入框”这个特定场景的必杀技
你担心的“输入框交互”其实是个特例。在输入框里打字，如果每个字都走一遍 后端 -> 广播 -> 前端，那打字会卡顿。

这里有一个行业标准做法：聚焦态 = 临时状态。

临时状态（前端私有）：
当你点击输入框开始输入时，这时的文字内容是纯粹的前端临时状态。它不属于“图层数据”，它只属于“当前输入框组件”。
这时，你随便改，不需要通知后端，也不需要存入全局 Data。
这就是为什么 Vue 组件内部可以有 data()，就是干这个用的。
提交状态（确认真理）：
当你输入完毕，点击“保存”、“失焦”或者按“Enter”时。
触发指令发给 C# 后端。
后端更新数据。
后端广播更新。
前端收到广播，把组件里的临时状态覆盖，正式显示在图层上。
这就解决了一切：

输入时：极快，因为完全是前端内部操作（Vue 管辖）。
保存后：绝对一致，因为是从后端拉回来的真理。
刷新页面：未保存的临时状态丢了（这是合理的），已保存的真理从后端拉回来了。
四、 最终架构建议
既然你有 C# 后端和 Electron，我强烈建议你采用 “本地 C/S 架构”：

C# 后端：
它是真理。
它维护一个内存中的 Document 对象。
它提供 WebSocket 或 IPC 接口。
它只接受“指令”，并在处理完指令后，推送“新状态”。
Vue 前端：
它是视图。
它维护一个 ViewData 对象（缓存）。
它只读这个对象来渲染界面。
用户的操作被封装成 Action 发给 C#。
它监听 C# 的推送，更新 ViewData。

// Vue 组件里

        // 1. 乐观更新：为了不卡顿，先让输入框显示出来
        this.localInputCache = newText; 
        
        // 2. 防抖发送指令给后端（不要每个字都发）
        debounce(() => {
            // 发指令，不是发数据
            backend.send({ cmd: 'updateLayerText', id: 123, text: newText });

// 全局 Store 里
mounted() {
    // 监听后端广播
    backend.on('layerUpdated', (data) => {
        // 用后端的真理，覆盖前端的缓存
        this.store.updateLayer(data);
    });

Vue 中 key 复用的技术分析：

Vue 如何处理 key：
在同一个父节点内部，如果 key 保持不变，Vue 会复用 VNode 和 DOM 元素。
当跨父节点移动时（从标签组 A 的子级变为标签组 B 的子级），Vue 的 Diff 算法流程会变得复杂。
旧版 Vue (Vue 2) 与 Vue 3：Vue 3 的 Diff 算法健壮得多，能更好地处理移动操作。
在 Vue 3 中，如果你将一个带有 key 的元素从列表 A 移动到列表 B，Vue 会高效地移动 DOM 节点并保持组件实例存活。这就是 Vue 3 的“区块树”和虚拟 DOM 优化的核心能力。
那为什么我之前说有“风险”或“坑”呢？
瞬态 DOM 状态： 即使 Vue 保留了组件实例，实际的 DOM 节点在浏览器中也是被物理移动的。<input> 值通常跟随 DOM 节点（除非使用了 v-model），但像文本选区（光标位置）、高亮状态或 :focus 等状态通常会在 DOM 移动/重新挂载时丢失。<input>` 值通常会保留，但聚焦状态绝对会丢失。
*生命周期钩子：* 如果 Vue 判断为移动，mounted/unmounted 钩子不会被触发，但 updated 会触发。如果代码逻辑依赖于 mounted 来初始化依赖于父级上下文的某些内容，那么这些逻辑可能无法正确触发。
*CSS 过渡/动画：* 移动父级内部的 DOM 节点通常具有清晰的过渡效果。将 DOM 节点从父级 A 移动到父级 B 在视觉上可能会显得突兀，或者会打断正在运行的动画。

或者，更有可能的是，他们**还没有实现它**，只是想：“如果我加上 key，它应该能工作吧？”

*策略：*
1.  **“同一父级”陷阱：** 解释 `key` 主要用于同一列表内的排序。跨父级取决于实现。
2.  **“销毁陷阱”：** 如果父级 A 和父级 B 是由不同数据数组驱动的不同组件，移动数据（从 A 移除，添加到 B）通常会导致销毁 -> 创建。
3.  **“状态”陷阱：** 如果它*确实*有效（例如，通过 Vue 3 的 Fragment/Teleport 魔术或单一扁平列表渲染），组件实例保留了。这很好，**但是**你跳过了生命周期钩子（`mounted`）。如果新父级需要不同的初始化，你的应用就会崩溃。
4.  **聚焦/光标问题：** 即使组件被复用，DOM 移动通常会清除文本输入的焦点/光标位置。

这是最有帮助的回答：告诉你 `key` 跨父级不可靠，并指出副作用。
这是一个非常敏锐的技术直觉问题。你可能会觉得：“只要 Key 一样，Vue 就能认出它是同一个组件，状态就能保留，多省事啊！”

答案是：在跨父节点（从标签组A拖到标签组B）的场景下，依赖 key 来复用组件是极其危险的，甚至是不可用的。

这不仅仅是“风险”问题，而是 Vue 的底层运作机制决定了它在这种场景下大概率不起作用。

我们分三个层面来戳破这个幻想：

1. Vue 的 Diff 算法机制（为什么它大概率会失效？）
Vue 的更新逻辑是以“父节点”为单位进行的。

当你把数据从 GroupA 移动到 GroupB 时：

GroupA 更新：Vue 发现少了一个子节点。它的默认行为是 销毁 这个组件实例（触发 onUnmounted），并从 DOM 中移除。
GroupB 更新：Vue 发现多了一个子节点。它的默认行为是 创建 一个新的组件实例（触发 onMounted），并插入 DOM。
那 key 呢？
key 的主要作用是在同一个父节点内部进行列表对比时使用的。

比如 v-for 在同一个列表里排序，Vue 看到 key 一样，会复用 DOM。
但是！ 跨父节点移动时，Vue 的 Diff 算法通常是分头进行的。GroupA 不会告诉 GroupB：“哎，我把那个 key=123 的节点传给你了，你别新建了，复用一下。”
结果： 即使你给了相同的 Key，99% 的情况下，Vue 依然会执行 “先销毁，后创建” 的过程。你的输入框内容依然会丢失，你的数据依然会重新拉取。

2. 即使强行实现了复用，会有什么副作用？（假设我们用了 Teleport 或黑科技）
假设你用了某种黑科技（比如 Vue 3 的 <Teleport> 或者特殊的移动逻辑），让组件实例真的没有被销毁，仅仅是移动了位置。这会带来更头疼的问题：

A. 生命周期钩子“欺骗”
组件移动了位置，但 onMounted 不会再次触发。
如果你的 Chat.vue 里有类似 onMounted(() => { fetchMessages() }) 的逻辑，它不会执行。
如果你的新位置需要不同的初始化逻辑（比如根据不同的标签组显示不同的标题），这些逻辑都不会跑。
B. 父级上下文丢失
Vue 组件是依托于父组件环境的。
如果 GroupA 提供了 provide: { theme: 'dark' }，而 GroupB 提供了 provide: { theme: 'light' }。
如果你把组件从 A 移到 B 且没有销毁，这个组件依然会“错误地”保持 A 的主题，因为它不会重新注入 B 的依赖。这会导致样式错乱、功能异常。
C. 焦点丢失
当 DOM 节点在 DOM 树中被物理移动位置时，浏览器引擎会强制触发一次“重排”。
浏览器的行为是：一旦 DOM 节点被移动，当前焦点会丢失。
这意味着，即使 Vue 组件的状态（内存数据）没丢，用户正在输入的光标也会瞬间没影了。这在体验上依然是不可接受的。
3. 正确的“复用”姿势：状态与视图分离
你想要 key 复用的本质目的，是不想丢失“正在编辑的数据”。

既然 Vue 在跨父节点移动时，必然（且应该）销毁重建组件，那我们就不应该让 Vue 去背负“保存数据”的责任。

解决方案对比：

方案	做法	结果
错误的 Key 复用	期待 Vue 不销毁	失败。Vue 会销毁。或者强行不销毁导致 Bug。
你现在的做法	数据存组件内	失败。组件一销毁，数据就没了。
正确的做法	数据存组件外	成功。组件销毁，数据还在外部 Store 里。新组件重建，直接读取。
具体操作：

不要把宝压在 Vue 的 key 上，而是要把 Chat.vue 写成“无状态”的样子：

组件重建：没关系，让它销毁，让它重建。

draftStore（全局变量）里存着 { 'tab-id-101': '这是用户输入的一半文字' }。

总结

1. Vue 官方文档里，key 的“官方定义”是什么？
Vue 文档里对 key 的说明有两段非常关键：

The key special attribute is primarily used as a hint for Vue’s virtual DOM algorithm to identify vnodes when diffing the new list of nodes against the old list.
vuejs.org

和

Children of the same common parent must have unique keys. Duplicate keys will cause render errors.
vuejs.org

重点：

“新列表 vs 旧列表”——指的是同一个父节点下的列表。
“Children of the same common parent”——同一个父节点的子节点之间 key 必须唯一。
再看列表渲染那一节：

When Vue is updating a list of elements rendered with v-for … This default mode is efficient, but only suitable when your list render output does not rely on child component state or temporary DOM state (e.g. form input values).
To give Vue a hint so that it can track each node’s identity, and thus reuse and reorder existing elements, you need to provide a unique key attribute for each item.
vuejs.org

这些话的潜台词是：

同一父节点内的列表，有 key → 可以复用/重排节点；
没有承诺“不同父节点之间，同一个 key 也能复用实例”。
所以你问：

它为什么要存个 key？

答：是为了让 Vue 在“同一父节点的子节点列表”里，能正确识别“哪些是同一个节点”，从而复用 DOM/组件状态。
不是为了跨父节点移动组件。

2. 跨父节点移动时，key 会不会帮你“保住组件实例”？
现实情况比较残酷：

2.1 Vue 的 diff 是按“父节点分头更新”来的
典型流程（简化）：

标签组A 的 vnode 走自己的 diff；
标签组B 的 vnode 走自己的 diff；
它们之间不会互相问：“我这边有个 key=123 的孩子被删了，是不是跑到你那边去了？”
所以：

从 A 的数组里删掉一项 → A 的更新逻辑：“这个 key 没了，销毁它”；
往 B 的数组里加一项 → B 的更新逻辑：“新增一个 key=123 的孩子，创建实例”。
结果就是：组件先被销毁，再被重建，即使 key 相同。

你可能会说：“但我本地跑起来，感觉没销毁啊？”那通常是：

你的实现方式其实没有真正跨父节点移动 vnode；
或者因为过渡动画、过渡模式、过渡 CSS 的巧合，看起来没销毁，但生命周期已经跑过 unmount/mount 了。
2.2 真正“跨父节点保实例”的是 Teleport，不是 key
Vue 3 提供的 <Teleport> 做的事情是：

把组件的 DOM 挪到别的 DOM 位置，但组件实例仍然挂在原来的父组件上。
vuejs.org

也就是说：

组件的“逻辑父节点”不变；
只是“DOM 父节点”变了。
这是官方设计好的能力，跟“靠 key 自己瞎搬”完全不是一回事。

4. 如果你硬要这么干，会遇到哪些“坑”？
假设你真的靠某种黑科技，让 Vue 不销毁实例，只移动 DOM，会遇到几个非常恶心的问题：

4.1 生命周期会“骗你”
移动到新的标签组后，onMounted 不会再触发；
如果你的 Chat.vue 里有 onMounted(() => fetchMessages())，它不会跑；
你会以为“聊天记录已经加载了”，其实根本没有，或者用的是旧数据。
4.2 provide/inject 上下文会错位
假如标签组A 提供了 theme: 'dark'，标签组B 提供了 theme: 'light'；
你把组件从 A 搬到 B，却没让组件重新走一遍 setup / created；
那组件内部 inject('theme') 拿到的还是 A 的主题，UI 样式就乱了。
4.3 焦点、滚动状态会丢
即使 Vue 组件实例没销毁，浏览器在 DOM 层面移动节点时，会丢失焦点和滚动位置；
你输入了一半的文字，在 DOM 层面被移走后，输入框会失焦，光标没了；
你要自己额外写逻辑去恢复焦点和滚动，这比“直接把状态提到外面”麻烦多了。

// stores/drafts.js

const props = defineProps(['tabId']);

const draft = computed({
